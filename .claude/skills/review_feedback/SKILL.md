# review_feedback

一定期間にたまったレビューサイクルをまとめて振り返り、レビューガイドラインの改善につなげるスキルです。

## 前提・思想

- **手動起動**: 自動実行はしません。完了サイクルが3〜5件たまった頃、または月1回程度を目安に、ユーザーが起動したときだけ動きます。
- **採用/棄却の一次情報は diff**: 「どの提案が採用されたか」は、draft.md（原稿）・review_*.md（提案）・final.md（公開版）の突き合わせで推定します。ユーザーに評価記録の手間を求めないための設計です。ただし `notes.md` がある場合は、diff からの推定よりそちらの明示的な所感を優先します。
- **human-in-the-loop**: ガイドラインの書き換えは必ず**修正差分案を提示し、ユーザーが承認してから**反映します。勝手に `guidelines/review_guideline.md` を編集しないでください。
- **「核 / 彩り」2層モデル**: 核（構成維持・事実の保全・全文リライト方式・著者の声）は安易に動かさず、彩り（レビューの重点・表現の好み）は積極的に調整します。この非対称が要です。
- **並走期間の比較**: `review_gemini.md` と `review_claude.md` が両方あるサイクルでは、両者の比較（指摘の重なり・採用率の差）も集計します。Claude への一本化を判断する材料になります。

## 手順

### ステップ0: 対象期間の決定

1. [meta/review_markers.md](../../../meta/review_markers.md) の最新エントリを確認する。
2. **dateFrom** = 最新エントリの対象期間末日の翌日、**dateTo** = 今日（JST）。エントリが1件も無い（初回）場合は、`reviews/` の最古のサイクルの日付を dateFrom にする。
3. 決めた期間をユーザーに一言示してから次へ進む。

### ステップ1: 対象サイクルの収集

1. `reviews/` 直下で、ディレクトリ名の日付が期間内のサイクルを列挙し、各ファイル（draft / review_gemini / review_claude / final / notes）の有無を表にする。
2. `final.md` が無いサイクルは**未完了**として集計対象から外す（次回に持ち越し。マーカーの持ち越し件数に記録する）。
3. 完了サイクルが0件なら、その旨を伝えて終了する（マーカーは更新しない）。

### ステップ2: 採用分析（サイクルごと）

各完了サイクルについて:

1. draft.md / review_gemini.md（あれば）/ review_claude.md（あれば）/ final.md を読み込む。
2. レビューの指摘・リライト提案のそれぞれについて、final での扱いを推定する: **採用**（ほぼそのまま反映）/ **部分採用**（形を変えて反映）/ **棄却**（原稿のまま、または別の形に修正）。
3. `notes.md` があれば、diff からの推定より優先して解釈に反映する。
4. 指摘を**カテゴリ分け**して整理する。カテゴリはガイドライン セクション2（彩り）の観点名を基本とし、当てはまらないものは新カテゴリ候補として控える（例: 冗長表現の圧縮 / 描写の具体化 / 語順・接続 / 構成 / 事実関係の確認 / 語彙選択）。

### ステップ3: 傾向集計

- **カテゴリ別の採用率**（全サイクル合算。件数と割合の表）。完了サイクルが3件未満の場合はサンプルが小さい旨を明記し、参考値として扱う。
- **並走比較**（review_gemini と review_claude が両方あるサイクルのみ）: 同一 draft への指摘の重なり / カテゴリ別採用率の差 / Gemini は指摘したが Claude が逃した指摘（およびその逆）。
- **繰り返し棄却されるパターン** = ガイドラインで抑制・修正すべき候補。**繰り返し採用されるパターン** = 強化・明文化すべき候補。1回だけの事象より、複数サイクルで繰り返されたものを優先する。

### ステップ4: 核 / 彩りの仕分け

ステップ2〜3の結果を、ガイドラインの2層へ仕分ける。

- **核への違反**（構成を変えてきた、事実を書き換えた、リライトが部分的だった、文体が別人になった等）→ 核を**締める**方向の文言修正。
- **彩りへの調整** → 2方向がある。
  - **抑制**: 繰り返し棄却される観点の重みを下げる／表現を限定する。
  - **強化**: 繰り返し採用される観点を明文化する／重みを上げる。
- **衝突判定**: 2つの傾向が本当に矛盾するのは**同じ層の同じ観点**に相反する結果が出たときだけ。その場合のみ、どちらを採るかの判断ポイントとしてユーザーに明示する。

### ステップ5: ガイドライン修正差分案の提示（承認 → 反映）

1. `guidelines/review_guideline.md` への具体的な修正差分案を before/after で提示する。各案には**根拠**（ステップ2〜3の採用率・件数・該当サイクル）を添える。
2. **ユーザーが承認した項目だけ** Edit で反映する。未承認・保留の項目は反映しない。
3. 反映した場合は、ガイドラインの変更履歴に日付と要約を追記する。

### ステップ6: マーカーの追記

[meta/review_markers.md](../../../meta/review_markers.md) の説明文直下（既存エントリの上）に、同ファイル記載のエントリ形式で1件追記する。

## 出力（ユーザーへの報告）

1. 対象期間と対象サイクル数（完了 / 未完了持ち越し）。
2. カテゴリ別採用率の表と、繰り返し現れた採用・棄却パターン。
3. 並走比較の結果（該当サイクルがある場合のみ）。Claude 一本化の判断材料になる所見があれば添える。
4. 核 / 彩りの仕分け結果（衝突があればその判断ポイント）。
5. 提示したガイドライン修正案と、承認・反映の有無。
6. 追記したマーカーの内容。
