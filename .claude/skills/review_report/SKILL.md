# review_report

登録済みの草稿（draft.md）に対して Claude がレビューを実行します。並走期間中は Gemini レビューのシャドーとして結果を残すだけで構いません（比較・評価はメタレビューが行います）。

## 参照してよいもの（コンテキスト分離）

- [guidelines/review_guideline.md](../../../guidelines/review_guideline.md)（必須）
- [guidelines/style_profile.md](../../../guidelines/style_profile.md)（蒸留済みの場合）
- 直近の公開版レポート2〜3本（`reviews/*/final.md` の新しいものから。ここでの新しさは山行日基準。足りなければ `corpus/` の新しいものを補充）

**上記以外は読まない。** 特に、過去サイクルの `review_gemini.md` / `review_claude.md` / `notes.md`、および `corpus/` の残り全体は参照しないこと（CLAUDE.md「参照の原則」）。過去のレビュー結果への追従ではなく、ガイドラインだけを根拠にレビューすることで、メタレビューによる改善ループを機能させます。

## 手順

1. **対象サイクルの特定**: 引数でサイクルディレクトリが指定されていればそれを使う。指定がなければ、`reviews/` 直下で最新（ディレクトリ名の日付降順。日付は山行日を表す）かつ `review_claude.md` 未作成のものを候補として提示し、ユーザーに確認する。`draft.md` を Read する。
2. **重点ポイントの確認**: ユーザーから「レビュー時に重点的に見てほしいポイント」「自分が気にしているポイント（うまく文章が組み立てられない、表現がうまくできない等の悩み）」といったコメント（任意）があるか確認する。
   - 引数や会話でコメントが渡されていればそれを使う。`draft.md` の frontmatter に `review_focus` があればそれも読み込む。
   - どちらも無い場合は、レビュー開始前に一度だけ「重点的に見てほしいポイントや気になっている点はありますか？（任意。無ければこのまま進めます）」と確認する。無ければそのまま進める。
3. **参照資料の読み込み**: 上記「参照してよいもの」を読み込む。style_profile.md が未蒸留の場合はその旨を認識し、直近の公開版から文体を読み取る。
4. **レビューの生成**: ガイドラインに従う。特に核（不変層）は必ず守る。
   - セクション構成・順序を変えない
   - 山行の事実（日付・ルート・山名・天候・所要時間・出来事）を変更・創作しない。疑義があれば指摘一覧で確認を促す
   - 表現の細部に踏み込む「ほぼ全文リライト」を行う
   - 著者の語り口（文体プロファイル）から外れない
   - ユーザーからの重点ポイント・悩みのコメントがあれば、レビューにあたっての考慮事項とする（該当箇所を重点的に確認し、指摘一覧とリライト案で具体的に応える）。ただし核（不変層）やガイドラインを上書きするものではない
5. **出力の構成**: ガイドライン セクション3の形式に従う。
   1. 総評（3〜5行。重点ポイントのコメントがあった場合は、それにどう応えたかにも触れる）
   2. 指摘一覧（番号付き。対象箇所の引用 / 理由 / 修正提案 / 根拠となるガイドラインの観点名）
   3. 全文リライト案（セクション構成維持）
6. **保存**: frontmatter（`reviewer: claude` / `reviewed_at` / `guideline_updated_at`: ガイドライン変更履歴の最新日付。重点ポイントのコメントがあった場合は `review_focus` にその要約も記録）を付けて `review_claude.md` として Write する。
7. **報告**: 指摘件数と要点のみ簡潔に報告する（全文はファイル参照とする）。
